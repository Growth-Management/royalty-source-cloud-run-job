# PODデータ加工仕様

## 目的

手順書「7 PODデータの加工」を、Cloud Run JobとBigQueryで再現する。

MS AccessのPOD販売報告テーブルを置換し、初期累積履歴と月次差分を同じAccess互換形式へ統合する。

## 確定した入力構成

### 初期移行正本

- ファイル: `⑤POD販売報告データ.xlsx`
- 内容: MS Accessへ2026年2月分まで取り込んだ後の累積テーブルのエクスポート
- 対象期間: 2014年3月から2026年2月
- 取込回数: 一度だけ
- 用途: BigQuery初期履歴

初期移行では、`【取込み用】202601_AmazonPOD販売実績(ICE).xlsx`よりもAccessエクスポートを優先する。前者は加工過程と月次シートの確認資料として扱い、累積正本として重複ロードしない。

### 月次差分

2026年3月以降は次の2系統を取り込む。

1. 従来Amazon POD月次データ
2. `【PF】販売レポート_*.csv`

2026年2月のAccess結果では、従来Amazon POD 4行とPF販売レポート由来15行が統合されていたため、PF販売レポートだけに一本化しない。

## 推奨テーブル

### STAGING

- `staging_pod_access_history`
- `staging_amazon_pod_monthly`
- `staging_pf_sales_report`

### RAW

- `raw_pod_access_history`
- `raw_amazon_pod_monthly`
- `raw_pf_sales_report`

STAGINGとRAWでは業務計算を行わず、元文字列と次のメタデータを保持する。

- `source_file_id`
- `source_file_name`
- `source_sheet_name`
- `row_number`
- `target_month`
- `loaded_at`
- `run_id`（監査ログとの関連付けで保持可能にする）

### SOURCE

- `source_pod_sales_report`

MS AccessのPOD販売報告テーブル相当の共通形式を生成する。

## Access互換列

| Access列 | 内容 |
|---|---|
| 発行社 | 原則 `ICE` |
| プロダクトコード | 入力値または商品マスタ補完 |
| ISBN | 入力値または商品マスタ補完。STRINGで保持 |
| タイトル | 入力値または商品マスタ正式タイトル |
| 単価 | 税別販売価格 |
| 掛率 | 従来Amazon POD入力値を保持。PFは既存Access結果との比較で確定 |
| 冊数 | 販売数 |
| 正味金額 | 売上金額（税別） |
| 消費税 | 原則0 |
| 売上金額 | 売上金額（税別） |
| 製造原価 | 下記変換規則 |
| 15 | 固定値 `1.5` |
| 108 | 固定値 `108` |
| 頁数 | 入力値または商品マスタ補完 |
| DL年月 | 原則、販売対象月の前月 `YYYYMM` |
| 年月 | 販売対象月 `YYYYMM` |

## PF販売レポート変換規則

| Access列 | PF CSV列・計算 |
|---|---|
| 発行社 | `ICE` |
| ISBN | `ISBN`。指数表記や欠損時は商品マスタから補完 |
| タイトル | `書籍名`または商品マスタ正式タイトル |
| 単価 | `販売価格（税別）` |
| 冊数 | `販売数` |
| 正味金額 | `売上金額（税別）` |
| 消費税 | `0` |
| 売上金額 | `売上金額（税別）` |
| 製造原価 | `販売手数料（税別） + 印刷費（税別）` |
| DL年月 | 販売月の前月 |
| 年月 | 販売月 |

次の関係を品質チェックする。

```text
支払額 = 売上金額 - 製造原価
       = 売上金額 - 販売手数料 - 印刷費
```

`販売数 = 0`の行はAccess取込対象から除外する。ただしRAWには保持する。

## 従来Amazon POD月次データ

従来形式は、既存の月次POD販売実績の列をAccess互換列へマッピングする。

- 転記用シートと元月次シートを同時に取り込まない
- `請求書`など販売明細以外のシートを除外する
- 月次シートから販売冊数が0でない行をSOURCE化する
- プロダクトコード、ISBNはSTRINGで保持する

## 初期移行方式

初回のみ`⑤POD販売報告データ.xlsx`を`raw_pod_access_history`へロードし、`source_pod_sales_report`の2014年3月から2026年2月を生成する。

初期移行済みフラグまたはファイルID監査によって、再実行時の二重投入を防止する。

## 月次方式

対象月単位で冪等に処理する。

1. 対象月の従来Amazon POD月次データをRAWへロード
2. 対象月のPF販売レポートをRAWへロード
3. 両入力をAccess互換形式へ変換
4. 対象月のSOURCEを置換
5. 品質チェックを実行

SOURCEは対象月単位のDELETE+INSERT、またはCREATE OR REPLACE相当で再実行時に重複しないこと。

## 2026年2月再現テスト

2026年2月を回帰テストの基準月とする。

| 指標 | 期待値 |
|---|---:|
| 行数 | 19 |
| 冊数合計 | 38 |
| 売上金額合計 | 51,138円 |
| 製造原価合計 | 37,027円 |
| 従来Amazon POD行数 | 4 |
| PF販売レポート由来行数 | 15 |

PF販売レポートの販売数非0行15件は、Access結果と販売数、売上金額、製造原価が一致すること。

## 品質チェック

最低限、次を監査テーブルへ記録する。

- 対象月NULL・不一致
- 販売数、売上金額、販売手数料、印刷費、製造原価の型変換失敗
- PF支払額計算不一致
- 販売数0行がSOURCEへ混入
- 商品マスタ未一致
- プロダクトコード未補完
- ISBN未補完または不正桁
- 業務キー重複
- 対象月SOURCE 0件
- 2026年2月の期待件数・数量・金額差分

## 実装上の注意

- PF CSVはCP932、UTF-8 BOMなどを順に判定して読む。
- ISBN、プロダクトコード、書籍IDは数値型にしない。
- 元ファイルの列はRAWで欠落させない。
- 商品マスタは`ice_qb_aggregation`を参照し、このリポジトリ内で重複生成しない。
- 初期履歴と月次差分の境界は2026年2月末とし、通常月次処理は2026年3月から開始する。
